## 1. 当前方向调整

本项目从这一版开始，产品方向正式从“工作流编排 / 手工定义任务流”调整为“对话驱动的桌面自动化工作台”。

新的核心原则：

1. 用户通过聊天直接表达目标，不再以手工创建 workflow / step / task tree 作为主入口。
2. workflow、task、step、approval 保留为后端内部执行机制，不再作为默认的用户显式编排对象。
3. 系统要形成真正的 agent loop：
   感知界面 -> 理解目标 -> 规划下一步 -> 执行动作 -> 再次感知 -> 判断完成/纠错。
4. 高风险动作仍然保留确认与安全机制，但要嵌入对话流程，而不是把用户拉回独立流程设计界面。
5. 数据组织以 conversation / session 为中心，而不是以 workflow definition 为中心。

## 2. 本轮新增的两份修改意见

### 2.1 代码治理意见

优先解决这些明确问题：

1. 清理 `main_window.py` 中重复定义、静默覆盖和合并残留。
2. 统一 `/api/health` 与 UI 的字段契约。
3. 把 `/api/chat` 从“硬耦合 workflow + 写死 Ollama”改成真正可配置的 agent 入口。
4. 让后端启动失败时 fail fast，不再伪装成“已启动”。
5. 降低事件噪音，减少 `append_event()` 的额外数据库读取。
6. 把权限审计真正落库到 `permission_audit`。
7. 修正任务查询的 Python 过滤和 N+1 访问问题。
8. 提升感知和动作链路稳定性，避免 `find_text` 结果无法安全传递到点击动作。
9. 重新校准测试，让测试与实现重新一致。

### 2.2 产品重构意见

后续所有功能调整，都要围绕下面的产品目标：

1. 聊天界面是唯一主入口。
2. 用户不需要手工创建 workflow。
3. 系统自动拆解目标、执行、观察、重试和回报。
4. UI 默认只保留：
   - 对话区
   - 当前执行状态
   - 屏幕感知结果 / 当前上下文
   - 动作日志
   - 暂停 / 继续 / 终止 / 人工接管
   - 权限确认
5. workflow editor、task tree、节点配置、流程作者视图默认从主界面移除，只允许保留在高级/调试模式。
6. 所有自动化围绕 conversation/session 组织，用户进入会话即可看到完整执行过程并继续修正任务。

## 3. 目标架构

后续开发优先收敛到这六条主链路：

1. `chat / session`
2. `agent orchestrator`
3. `perception`
4. `desktop controller / system executor`
5. `memory / persistence`
6. `permission & safety`

不再继续优先投入这些方向：

- 手工编排工作流编辑器
- 通用 BPM / 流程引擎式产品形态
- 让用户维护节点、分支、审批图
- 以脚本编写作为主要使用方式

## 4. 重构阶段

### 阶段 G：稳定性治理与契约收口

目标：先把当前实现中的漂移、重复和契约错位清理掉，恢复一个可信的基础版本。

工作项：

1. 清理 UI 合并残留，拆掉重复方法定义，收敛为单一主界面实现。
2. 修复 `/api/health`、`/api/chat`、设置页、任务页之间的接口契约不一致。
3. 修复启动链路，让 `BackendServer.start()` 在服务未就绪时明确失败。
4. 优化事件系统，减少无意义事件和重复数据库读。
5. 接通权限审计落库。
6. 优化任务列表与会话任务查询，避免 Python 过滤和 N+1。
7. 修复测试漂移，先恢复核心后端测试可信度。

验收标准：

- UI 不再存在重复定义覆盖。
- 健康检查、聊天、任务详情的前后端字段一致。
- 后端启动失败时会中断启动流程。
- 权限动作可以在数据库中审计。
- 核心测试可以真实反映当前实现状态。

### 阶段 H：对话式 Agent 工作台重构

目标：把产品重心彻底切换到“聊天即任务、任务围绕会话、workflow 对用户隐身”。

工作项：

1. 把聊天入口改造成真正的 agent 入口，而不是“聊天 + 单次工具调用”。
2. 建立 conversation-centered agent loop。
3. 将内部 workflow 退居为 orchestrator 的执行状态机。
4. 默认 UI 只保留对话、状态、上下文、日志、确认与人工接管。
5. 将高风险确认嵌入对话和执行流，不再跳回独立设计界面。
6. 支持失败重试、卡点说明、继续会话修正任务。

验收标准：

- 用户只通过聊天就能发起并完成一个桌面任务。
- 用户不需要手工创建 workflow。
- 系统能连续执行多个感知-动作回合。
- 高风险动作会暂停并请求确认。
- 失败时会自动重试或向用户说明卡点。
- 用户能在一个会话中看到完整执行过程并继续修正任务。
