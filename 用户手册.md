# 用户手册

最后更新：2026-04-13

## 0. 维护约定

- 每次修改后都要同步更新 `checklog.md`
- 每次影响产品入口、交互方式或能力边界的修改，都要同步更新本手册

## 1. 产品定位

`Agent Studio` 当前是一套“对话驱动的桌面自动化工作台”。

默认使用方式是：

1. 在聊天区直接说目标
2. 系统在会话内部创建并推进自动化任务
3. 右侧区域只展示当前会话下的 Agent 状态、审批和事件

你不需要先手工创建工作流，也不需要先配置一整套步骤。

## 2. 启动方式

在项目根目录运行：

```powershell
.\scripts\run.ps1
```

如果只想检查后端是否在线，可以访问：

- `GET /api/health`

## 3. 主界面说明

### 左侧：会话与聊天

左侧是主入口，包含：

- 会话列表
- 当前会话聊天历史
- 图片附件入口
- 目标输入框
- `Send`

推荐使用方式：

1. 直接在输入框描述你要完成的目标
2. 如果有需要，附上一张或多张图片
3. 点击 `Send`

### 右侧：当前会话执行态

右侧只展示当前会话相关内容，不再要求你手工编排流程。

当前会显示：

- 当前会话下的执行标签
- 每次执行里的 Agent / Subagent 标签
- 执行状态
- 最近事件
- 高风险动作的内联确认区

界面上会优先展示“状态 + 标题”的执行标签，而不是直接暴露内部 task ID。

从这个版本开始，左右两侧有明确映射入口：

- 左侧每条已关联执行的消息会显示“对应执行”链接，点击后直接跳到右侧对应执行标签
- 右侧执行摘要会显示“触发消息”链接，点击后会反向定位并高亮左侧触发消息
- 如果没有显式映射字段（`linked_task_id` 或 `source_message_id`），界面会显示“无映射”，不会猜测绑定关系
- 每个会话都会显示专属 `Sandbox` 目录，附件会进入该目录，不会直接复用任意历史本地路径

## 4. 聊天即任务

当前版本里，聊天已经接入会话内的自主任务执行。

每次发送目标时，系统会：

1. 先确保当前会话存在
2. 在当前会话里创建或推进一个自主任务
3. 让 Agent 根据目标自行规划下一步
4. 把结果写回当前会话

重要变化：

- 聊天任务现在默认继承当前设置页里的 `provider / model / base_url`
- 不再写死成本地 `Ollama`
- `/api/chat` 仍然优先走 autonomous task，但当 planner/执行失败或返回低信号占位文案时，会自动降级到 direct chat，保证聊天入口可用
- 发生降级时，会话历史仍会保留 task 关联：如果 task 已创建，assistant 消息会继续绑定该 task
- 如果当前 provider 允许 fallback，底层模型路由仍然可以降级到更安全的备用路线
- 右侧执行区现在更偏向“会话执行快照”，弱化了 workflow 式词汇

新增会话治理能力：

- 支持删除会话（UI 中的 `Delete Conversation`）
- 删除后会从数据库中移除该会话及其消息、并清理该会话关联的任务记录
- 会话沙箱目录会在删除时一起清理

## 5. 图片附件

聊天区支持附图。

你可以：

- 点击 `Add Images` 选择本地图片
- 点击 `Clear Images` 清空当前附件

图片会作为当前会话的视觉上下文供模型和 Agent 使用。

## 6. 设置页

右上角 `Settings` 可以配置：

- 主模型 Provider
- Model
- Base URL
- API Key
- Organization
- 桌面控制模式
- 审批超时时间
- 审批超时默认动作
- 超时额外提示词
- 脚本审查模型设置
- 界面语言
- 输出细节模式

设置页也支持：

- 单独测试当前 Provider 连通性
- 一次性检查全部 Provider 连通性
- 查看各 Provider 能力摘要

## 7. 高风险动作确认

高风险动作不会静默执行，而是以内联确认的方式出现在主界面右侧。

当前确认区支持三种操作：

- `Allow`
- `Deny`
- `Extra Prompt`

同时支持：

- 60 秒倒计时
- 超时后的默认动作
- 在设置页切换默认同意 / 默认拒绝 / 默认提示词

脚本审查会优先使用单独配置的审查模型，不必和执行任务的主模型相同。

## 8. 当前支持的核心能力

系统内部已经具备这些能力：

- 多模型路由
- 本地多模态
- 会话持久化
- 截屏
- OCR
- 文本定位
- 桌面控制授权
- Agent / Subagent 递归执行
- 高风险脚本审查与批准执行
- 全量 Provider 连通性检查

这些能力现在更多作为系统内部执行机制存在，而不是要求你手工操作它们。

## 9. 当前实现说明

当前版本还有几条重要实现原则：

- workflow / task / step 仍然存在，但主要用于内部执行状态记录
- 后端启动如果未就绪，会直接 fail fast，而不是伪装成“已启动”
- 桌面控制决策会写入 `permission_audit`
- 会话相关状态、最近事件和任务状态都会持久化保存
- 中英文界面文案已经重新清理为统一 UTF-8 文本，避免此前的编码污染

## 10. 当前限制

当前版本仍处于“工作流原型”向“对话式 Agent 工作台”过渡阶段，所以还有限制：

- 右侧虽然已经弱化 task 术语，但内部执行结构还没有完全对普通用户隐身
- workflow 内部结构还没有完全对普通用户隐身
- 更完整的连续 agent loop、自动纠错和人工接管流程仍在后续阶段
